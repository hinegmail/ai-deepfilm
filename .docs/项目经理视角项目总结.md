## 一、项目概况

### 1.1 基本信息

| 项目 | 信息 |
|------|------|
| **项目名称** | AI 短剧工作室（AI Manga Studio / Print Film） |
| **版本号** | 0.1.0（package.json） / v2.1.1（文档标注） |

### 1.2 项目目标

构建一个 **不依赖任何后端服务器** 的 AI 短剧/漫剧创作工作台，通过浏览器本地存储和 OpenAI 兼容 API 实现从剧本到视频的完整创作流程。

### 1.3 项目范围

```
项目边界
├── 包含
│   ├── 剧本创作与 AI 拆解
│   ├── 角色定妆/场景概念图生成
│   ├── 关键帧驱动视频生成
│   ├── 视频预览与导出
│   ├── 多模型提供商配置系统
│   └── Web/桌面/Docker 三形态部署
│
└── 不包含
    ├── 视频剪辑（拼接/特效/字幕）
    ├── 音频生成与编辑
    ├── 多人协作与云同步
    ├── 用户账号系统
    └── 自建 AI 推理引擎
```

### 2.4 API 适配器设计

产品采用 **适配器模式** 隔离不同 AI 提供商的 API 差异：

| 适配器 | 接口 | 兼容格式 | 特殊处理 |
|--------|------|---------|---------|
| chatAdapter | `/chat/completions` | OpenAI Chat | JSON 清理、重试机制、超时控制 |
| imageAdapter | `/images/generations` 或 `/chat/completions` | OpenAI Image + Gemini | 参考图注入、Base64 提取、多格式兼容 |
| videoAdapter | `/videos` 或 `/chat/completions` | Sora/Veo/豆包 | 异步轮询、图片缩放、多模式分流 |

**关键设计决策**：
- 三级 API Key 优先级：模型 Key > 提供商 Key > 全局 Key
- 本地服务自动绕过 Key 校验（返回占位符）
- 本地开发时统一走 `/api-proxy` 代理避免 CORS
- 视频适配器支持同步/异步/豆包三种模式

## 五、架构设计评估

### 5.1 架构亮点

#### 1. 适配器模式隔离 AI 差异

```
modelService（统一入口）
    ├── chatAdapter → callChatApi() → fetch(/chat/completions)
    ├── imageAdapter → callImageApi() → fetch(/images/generations)
    └── videoAdapter → callVideoApi()
                        ├── callVeoApi()（同步）
                        ├── callSoraApi()（异步轮询）
                        └── callDoubaoSeedanceApi()（豆包）
```

**价值**：上层业务逻辑不感知具体 API 差异，新增模型只需配置无需改代码。

#### 2. 三级 API Key 优先级

```
模型专属 Key > 提供商 Key > 全局 Key > 本地占位符
```

**价值**：灵活适应从单 Key 全局使用到每模型独立 Key 的各种场景。

#### 3. 三层数据迁移防护

```
Layer 1: loadRegistry() 加载时过滤废弃提供商
Layer 2: getProviders() 返回时再次过滤
Layer 3: UI 层只显示 isBuiltIn 的提供商
```

**价值**：确保旧版本用户升级后不会看到已废弃的提供商（如 GitCC）。

#### 4. 多部署形态统一代理

```
开发环境:  Vite proxy      → /api-proxy → AGNES AI
桌面应用:  Express proxy   → /api-proxy → AGNES AI
容器部署:  Nginx proxy     → /api-proxy → AGNES AI
```

**价值**：前端代码无需感知部署形态，统一通过 `/api-proxy` 访问 AI API。

### 5.2 架构风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| IndexedDB 容量限制（通常 50MB-数 GB） | 大量视频/图片存储可能超限 | 中 | 建议增加导出功能+容量警告 |
| localStorage 容量限制（5-10MB） | 模型配置过多可能超限 | 低 | 当前数据量远低于限制 |
| 浏览器清除数据导致项目丢失 | 用户工作量全部丢失 | 高 | P0 增加项目导出/导入 |
| 上游 API 变更导致适配器失效 | 某些功能不可用 | 中 | 多提供商支持降低单一依赖 |
| Tailwind CDN 不可用 | 样式加载失败 | 低 | 考虑迁移到构建链 |

### 7.2 技术风险深度分析

#### 风险 1：数据持久化

**现状**：所有项目数据存储在浏览器 IndexedDB，无云端备份。

**影响**：
- 用户清除浏览器数据 → 所有项目丢失
- 浏览器存储空间不足 → 新数据无法保存
- 跨设备使用 → 无法同步项目

**建议**：
1. 短期：增加项目 JSON 导出/导入功能
2. 中期：增加 IndexedDB 容量监控和警告
3. 长期：考虑可选的云端同步服务

#### 风险 2：单体服务文件

**现状**：`geminiService.ts` 单文件 2700+ 行，包含剧本解析、分镜生成、视觉提示词、关键帧优化、视频生成等几乎所有 AI 调用逻辑。

**影响**：
- 维护困难，修改一处可能影响多处
- 无法独立测试单个功能
- 新功能添加导致文件继续膨胀

**建议**：
1. 按业务域拆分为独立服务（scriptService、visualPromptService、videoService 等）
2. 将通用 HTTP 调用逻辑提取到公共模块
3. 逐步迁移，保持向后兼容
